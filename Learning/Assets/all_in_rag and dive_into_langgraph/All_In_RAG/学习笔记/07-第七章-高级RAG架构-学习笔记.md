## 要点整理

### 1. 知识图谱基础
- **定义**：由实体（节点）和关系（边）组成的结构化知识表示
- **基本单元**：三元组 (Subject → Predicate → Object)
- **示例**：(北京, 是首都, 中国) | (姚明, 效力于, 火箭队)

### 2. 传统 RAG vs KG-RAG
| 维度 | 传统向量 RAG | 知识图谱 RAG |
|------|-------------|-------------|
| **知识表示** | 文本块（Chunk） | 实体-关系-实体（三元组） |
| **检索方式** | 向量相似度搜索 | 图遍历 + 结构匹配 |
| **推理能力** | 弱（依赖LLM隐式推理） | 强（显式关系路径推理） |
| **多跳问题** | 容易丢失中间步骤 | 天然支持多跳推理 |
| **可解释性** | 中 | 高（可回溯关系路径） |
| **构建成本** | 低 | 高（需实体和关系抽取） |

### 3. KG-RAG 检索流程
1. **实体识别**：从用户问题中识别关键实体
2. **图遍历**：在知识图谱中查找相关实体及其关联
3. **子图提取**：提取与问题相关的子图结构
4. **上下文生成**：将子图转换为文本描述
5. **LLM 生成**：基于图结构和文本上下文生成答案

### 4. 构建知识图谱的挑战
- **实体抽取**：准确识别文本中的命名实体
- **关系抽取**：正确判断实体之间的语义关系
- **知识融合**：合并不同来源的知识，处理实体对齐
- **质量保证**：确保三元组的准确性和完整性
- **持续更新**：知识图谱需要随知识变化不断更新

### 5. 适用场景
| 场景 | 适用性 | 原因 |
|------|--------|------|
| 多跳推理问答 | ⭐⭐⭐ | 图遍历天然支持关系链推理 |
| 关系密集型知识 | ⭐⭐⭐ | 如药物-疾病-副作用关系网络 |
| 需高可信度场景 | ⭐⭐⭐ | 可解释性高，可追溯推理路径 |
| 简单事实问答 | ⭐⭐ | 向量 RAG 也足够 |
| 开放域问答 | ⭐ | 构建范围难以覆盖 |

## 参考答案汇总

> ⚠️ 先自己作答，再对照检查

### 基础题答案

1. 知识图谱是以图结构组织和表示知识的数据模型，基本组成单元是实体（节点）和关系（边）。

2. 传统 RAG 用向量相似度搜索文本块，KG-RAG 用图遍历搜索实体和关系。KG-RAG 在多跳推理和可解释性方面更强。

3. 三元组是 (主体, 关系, 客体) 的知识表示形式。示例：(姚明, 出生于, 上海)。

4. KG-RAG 可解决：① 多跳推理能力不足；② 关系层面理解缺失；③ 检索可解释性差；④ 实体间关联不被充分利用。

5. Neo4j（图数据库）、ArangoDB（多模型数据库）、Amazon Neptune（云图数据库）。

### 进阶题参考答案

1. 常用方法：① 基于规则的模式匹配；② 基于预训练模型的 NER（命名实体识别）+ RE（关系抽取）；③ LLM 驱动的抽取（Few-shot Prompt 从文本中抽三元组）；④ 半自动化：模型初抽 + 人工审核。

2. 协同流程：先用向量检索做语义匹配找到相关文档片段 → 从片段中识别实体 → 在图谱中扩展遍历关联实体 → 融合向量和图检索结果。这种方式兼具语义理解能力和结构化推理能力。

3. Graph RAG 特别适合需要多步推理的问题，如"A公司投资的初创企业中，有哪些获得了B基金的投资？"——传统向量 RAG 需要通过文档级的语义匹配间接处理，容易丢失中间环节；而 Graph RAG 可以直接在图谱中沿着关系链遍历 (A公司 → 投资 → 初创企业 → 获得投资 → B基金)，确保每步推理都有显式的路径支持。

### 编程题参考实现

```python
from neo4j import GraphDatabase

# 连接 Neo4j
driver = GraphDatabase.driver("bolt://localhost:7687", auth=("neo4j", "password"))

# 创建实体与关系
def create_knowledge_graph(tx):
    tx.run("CREATE (a:Company {name: 'OpenAI'})")
    tx.run("CREATE (b:Product {name: 'ChatGPT'})")
    tx.run("MATCH (a:Company {name: 'OpenAI'}), (b:Product {name: 'ChatGPT'}) "
           "CREATE (a)-[:DEVELOPS]->(b)")

with driver.session() as session:
    session.execute_write(create_knowledge_graph)

# 图检索查询
def query_graph(tx, question):
    result = tx.run(
        "MATCH (a)-[r]->(b) "
        "WHERE a.name CONTAINS $entity OR b.name CONTAINS $entity "
        "RETURN a.name, type(r), b.name",
        entity=question
    )
    return [record for record in result]

with driver.session() as session:
    results = session.execute_read(query_graph, "OpenAI")
    for r in results:
        print(f"({r['a.name']}) -[{r['type(r)']}]-> ({r['b.name']})")
```

### 大厂面试题参考解答

**题目 1（百度）：**
Graph RAG 用图结构（实体-关系）表示知识，检索时通过图遍历找到相关的三元组。传统 RAG 用向量表示文本块，检索时通过语义相似度搜索。
选用 Graph RAG 的场景：多跳推理问题、"关系"密集型知识（如医疗/药物相互作用）、高可解释性要求的场景（如金融合规审查）。
选用传统 RAG 的场景：简单事实问答、大规模文档搜索、知识快速更新、构建成本敏感。

**题目 2（腾讯）：**
评估是否引入 Graph RAG 的三个标准：① **查询类型**：业务中是否存在大量多跳推理问题？② **知识结构性**：领域知识是否天然具有图结构（如组织架构、知识体系）？③ **ROI**：人工构建图谱的维护成本 vs 效果提升价值。通常建议先用传统 RAG 做 MVP，当发现有明确的多跳推理瓶颈时再引入 KG-RAG，也可使用 LLM 自动抽取实体关系降低构建成本。
