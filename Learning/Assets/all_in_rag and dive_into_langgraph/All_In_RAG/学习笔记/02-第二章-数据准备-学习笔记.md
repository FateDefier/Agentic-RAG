## 要点整理

### 1. 数据加载核心任务
| 任务 | 说明 |
|------|------|
| **文档解析** | 将 PDF、Word、Markdown 等提取为可处理文本 |
| **元数据抽取** | 抽取来源、页码、作者等关键信息 |
| **结构化统一** | 整理成统一数据结构，便于后续处理 |

### 2. 主流文档加载器对比
| 工具 | 适用场景 | 特点 |
|------|----------|------|
| PyMuPDF4LLM | 科研文献、技术手册 | PDF→Markdown，OCR+表格识别 |
| TextLoader | 纯文本处理 | 轻量高效 |
| DirectoryLoader | 混合格式文档库 | 批量处理，多格式扩展 |
| Unstructured | PDF/Word/HTML 等 | 统一接口，智能解析 |
| FireCrawlLoader | 在线文档、新闻 | 实时内容获取 |
| LlamaParse | 法律合同、学术论文 | 解析精度高，商业API |
| Docling | 企业合同、报告 | 模块化，IBM生态兼容 |
| Marker | 科研文献、书籍 | GPU加速，PDF→Markdown |
| MinerU | 学术文献、财务报表 | 多模态集成，LayoutLMv3+YOLOv8 |

### 3. 文本分块核心要点
- **嵌入模型限制**：输入长度上限（如 bge-base-zh-v1.5 为 512 token）
- **LLM 上下文限制**：检索块需能放入 LLM 上下文窗口
- **块不是越大越好**：
  - 信息损失：长块 → 向量压缩 → 关键细节模糊化
  - Lost in the Middle：LLM 倾向于记忆开头和结尾，忽略中间
  - 主题稀释：多主题混合 → 检索信号比降低

### 4. 基础分块策略
| 策略 | 方式 | 适用场景 |
|------|------|----------|
| **固定大小分块** | 按字符数分割 + 重叠 | 通用场景 |
| **递归字符分块** | 按段落/句子/词层级递归分割 | 结构化文档 |
| **语义分块** | 基于语义边界（如句号、段落）切分 | 高精度需求 |
| **文档结构感知分块** | 利用标题/章节结构 | 长文档/技术文档 |

## 参考答案汇总

> ⚠️ 先自己作答，再对照检查

### 基础题答案

1. ① 解析不同格式的原始文档，提取纯文本；② 抽取文档来源、页码、作者等元数据；③ 将文本和元数据整理为统一数据结构。

2. 详见要点整理第 2 点表格。

3. 文本分块是将加载后的长篇文档切分为更小、更易于处理的单元，这些单元是后续向量检索和 LLM 处理的基本单位。它是关键步骤，因为嵌入模型和 LLM 都有输入长度限制，且合理的分块能提高检索的精确度。

4. **嵌入过程**：文本块越长，单个向量承载的信息越稀释，关键细节被模糊化，降低检索精度。**生成过程**：过大的上下文会导致"Lost in the Middle"现象，LLM 更关注开头和结尾，忽略中间的关键信息。

5. 详见要点整理第 4 点表格。

### 进阶题参考答案

1. **Lost in the Middle**：当 LLM 处理非常长的上下文时，倾向于更好地记住开头和结尾的信息，忽略中间部分。缓解策略：① 合理分块，控制单块大小；② 对检索结果进行 Rerank，将最相关的内容放在开头或结尾；③ 优化 Prompt 设计，引导模型关注全部上下文。

2. **主题稀释**：多主题混合在一个块中 → 块的语义向量变得笼统 → 检索时无法精确匹配。
   差策略：将"技能介绍"+"推荐出装"+"背景故事"全放在一个大块中。查询"出装"时，整体的相关性得分被稀释。
   好策略：将三个主题分别切分。查询"出装"时，"推荐出装"块获得高相关性得分，被精准召回。

3. 数据加载质量直接影响后续所有环节。低质量加载（如 PDF 解析乱码、表格丢失）会导致分块产生噪音、向量嵌入失真、检索召回无关内容、LLM 生成错误答案。因此"垃圾进，垃圾出"（GIGO）是 RAG 系统的铁律。

### 编程题参考实现

```python
from langchain.text_splitter import CharacterTextSplitter, RecursiveCharacterTextSplitter
from langchain_community.document_loaders import PyMuPDFLoader

# 加载 PDF
loader = PyMuPDFLoader("document.pdf")
documents = loader.load()

# 方式1：固定大小分块
splitter1 = CharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks1 = splitter1.split_documents(documents)

# 方式2：递归字符分块（推荐）
splitter2 = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", "。", "！", "？", "，", " ", ""]
)
chunks2 = splitter2.split_documents(documents)

print(f"方式1: {len(chunks1)} chunks")
print(f"方式2: {len(chunks2)} chunks")
```

### 大厂面试题参考解答

**题目 1（字节跳动）：**
Chunk 策略需要结合嵌入模型的上下文窗口（通常 ≤512 token）和 LLM 的上下文窗口。
经验法则：chunk_size 从 256-1024 token 起步；chunk_overlap 设为 10%-20%。
常用方法：先用 RecursiveCharacterTextSplitter 按自然段落/句子分割，再根据 token 数调整。
语义分块（Semantic Splitter）是更优方案，能保持语义完整性。

**题目 2（腾讯）：**
常见挑战：PDF 多栏布局错乱、表格结构丢失、扫描件 OCR 识别、Word 样式解析。
解决方案：① 使用专业解析工具（如 Unstructured、PyMuPDF4LLM）；② 对扫描件增加 OCR 预处理；③ 使用多模态模型（如 MinerU 集成的 LayoutLMv3+YOLOv8）识别复杂布局；④ 质检环节，人工抽检解析结果。
