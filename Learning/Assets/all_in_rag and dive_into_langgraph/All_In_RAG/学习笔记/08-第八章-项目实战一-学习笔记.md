## 要点整理

### 1. 项目整体架构
| 模块 | 功能 | 关键组件 |
|------|------|----------|
| **环境配置** | 项目初始化+依赖管理 | Docker, Conda, requirements.txt |
| **数据准备** | 文档加载+文本分块 | LangChain Loaders, Text Splitters |
| **索引构建** | Embedding+向量存储 | HuggingFace Embeddings, FAISS/Milvus |
| **检索模块** | 语义搜索+结果排序 | 向量检索, Rerank |
| **生成模块** | 上下文组装+LLM调用 | Prompt模板, LLM API |
| **系统整合** | API服务+前端交互 | FastAPI, Streamlit |

### 2. 核心实现流程
1. **环境初始化**：配置虚拟环境，安装依赖
2. **数据加载**：支持 PDF/Word/Markdown/HTML 多格式
3. **文档分块**：RecursiveCharacterTextSplitter 层级分割
4. **向量嵌入**：使用 bge-small-zh-v1.5 等国产嵌入模型
5. **索引存储**：存入 FAISS（本地）或 Milvus（生产）
6. **检索召回**：语义搜索 + 自定义过滤
7. **生成回答**：构建 Prompt → LLM API → 结果解析

### 3. 关键技术栈
| 类别 | 技术选型 |
|------|----------|
| 框架 | LangChain / LlamaIndex |
| 嵌入模型 | BAAI/bge-small-zh-v1.5 |
| LLM API | DeepSeek / OpenAI / AIHubmix |
| 向量库 | FAISS（原型）/ Milvus（生产） |
| 文档解析 | PyMuPDF / Unstructured |
| 服务部署 | FastAPI / Streamlit |

### 4. 优化策略
- **检索优化**：添加 Rerank 提升 Top-K 准确率
- **Prompt 优化**：明确约束，添加无答案兜底
- **性能优化**：缓存、异步处理、批量检索
- **质量监控**：定期评估，收集用户反馈

## 参考答案汇总

> ⚠️ 先自己作答，再对照检查

### 基础题答案

1. 环境配置、数据准备、索引构建、检索模块、生成模块、系统整合（共六大模块）。

2. ① 多格式文档加载（PDF/Word/HTML/Markdown）；② 文档元数据抽取；③ 文本分块（确定 chunk_size 和 overlap）；④ 数据清洗和质量检查。

3. 开发阶段使用 FAISS（轻量本地化），生产部署使用 Milvus（分布式高性能）。FAISS 作为算法库无需运维服务，适合快速迭代；Milvus 支持分布式部署和 GPU 加速，适合大规模生产环境。

4. 文档加载 → 文本分块 → 嵌入向量化 → 索引构建（FAISS/Milvus）→ 索引持久化。

5. 将检索到的 Top-K 文档块作为上下文，与用户问题一起填充到预设的 Prompt 模板中，调用 LLM API 生成回答。

### 进阶题参考答案

1. 分块策略结合嵌入模型上下文窗口（通常 512 token）和文档结构。chunk_size=500（小于嵌入模型限制），chunk_overlap=50（10%）。使用 RecursiveCharacterTextSplitter，分隔符优先考虑段落（\n\n）和句子（。！？），尽量保持语义完整。

2. 使用统一的文档加载器接口（如 LangChain 的 DirectoryLoader），根据不同文件类型自动匹配对应的加载器。所有格式统一转为 Document 对象（含 page_content 和 metadata），再走相同的分块→嵌入→索引流程。

3. 常见瓶颈：① API 调用延迟 → 异步并发调用；② 向量检索慢 → 使用索引（IVF/HNSW）；③ 上下文太大 → 限制最大 token 数；④ 响应长 → 流式输出。

### 编程题参考实现

```python
# 完整 RAG 问答系统（简化版）
import os
from langchain_community.document_loaders import DirectoryLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.embeddings import HuggingFaceEmbeddings
from langchain_community.vectorstores import FAISS
from langchain.chains import RetrievalQA
from langchain_community.llms import OpenAI

# 1. 加载文档
loader = DirectoryLoader("./data/", glob="**/*.md")
docs = loader.load()

# 2. 分块
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500, chunk_overlap=50,
    separators=["\n\n", "\n", "。", "！", "？", "，", " "]
)
chunks = splitter.split_documents(docs)

# 3. 向量化 + 索引
embeddings = HuggingFaceEmbeddings(model_name="BAAI/bge-small-zh-v1.5")
vectorstore = FAISS.from_documents(chunks, embeddings)

# 4. 检索问答链
qa = RetrievalQA.from_chain_type(
    llm=OpenAI(model="deepseek-chat"),
    retriever=vectorstore.as_retriever(search_kwargs={"k": 3}),
    chain_type="stuff"
)

# 5. 交互
while True:
    query = input("\n请输入问题（输入exit退出）: ")
    if query.lower() == "exit":
        break
    answer = qa.invoke(query)
    print(f"\n答案: {answer['result']}")
```

### 大厂面试题参考解答

**题目 1（字节跳动）：**
LangChain 和 LlamaIndex 各有侧重。个人通常混用：LlamaIndex 专攻索引检索（数据处理和索引构建更专业），LangChain 负责编排对话流程（Agent、Chain 编排更灵活）。技术选型考虑：数据复杂度（是否需要多源索引）、检索策略（是否需要高级路由）、团队熟悉度。

**题目 2（阿里）：**
排查三步走：① 查检索是否准确——检查 Hit Rate、MRR 等指标，看召回的是否相关；② 查上下文质量——检查 Rerank 后进入 LLM 的文档是否相关、是否完整；③ 查生成质量——检查 Faithfulness 指标，看 LLM 是否忠实于上下文。
优化路径：检索端调 chunk 策略 → 加 Rerank → 优化 Prompt → 调 LLM 参数。

**题目 3（腾讯）：**
数据质量最重要。RAG 系统的效果上限取决于数据质量，80% 的效果问题来自数据准备环节（解析乱码、分块不合理、噪音数据）。其次是检索策略设计，再次是 Prompt 设计。建议将主要精力投入数据清洗和分块策略优化。
