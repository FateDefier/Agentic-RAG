## 要点整理

### 监督者模式核心概念

| 概念 | 说明 |
|------|------|
| **工作流程** | 用户 → Supervisor（任务拆解）→ 工具节点/工具 Agent（执行）→ Supervisor（结果汇总）→ 用户 |
| **核心思想** | 一个中央 Agent 负责规划和分发，多个专业 Agent 负责执行 |
| **控制流** | 集中式控制，Supervisor 掌握全局 |

### 两种实现方式对比

| 维度 | Tool-Calling 方式 | langgraph-supervisor 包 |
|------|------------------|----------------------|
| 实现原理 | 将 sub-agent 包装为 `@tool` | 使用 `create_supervisor` 函数 |
| 依赖包 | langchain | langgraph-supervisor |
| 灵活性 | 高，可精细控制每个工具 | 高，自动化程度更高 |
| 复杂度 | 需要手动包装 | 开箱即用 |
| 上下文管理 | 手动控制 | 自动处理 |

### 适用场景

| 场景 | 说明 |
|------|------|
| 工具过多 | 单个 Agent 工具太多时，容易在工具选择上出错 |
| 上下文过大 | 一个 Agent 无法有效跟踪大量上下文 |
| 专业化需求 | 不同子任务需要不同领域的专业知识 |

### 流程图

```
用户 → Supervisor
    ├─→ 工具节点（直接执行）
    ├─→ 工具 Agent 1（专业任务）
    └─→ 工具 Agent 2（专业任务）
    ↓
Supervisor → AI 回复
```

## 参考答案汇总

### 基础题答案

1. **监督者模式**：一个 Agent 充当监督者（Supervisor），负责将用户任务拆解成具体步骤，交给工具节点或工具 Agent 执行。工作节点将执行结果返回给监督者，再由监督者汇总结果返回给用户。核心是"一个中心化的主管 Agent 负责分配任务和汇总结果"。

2. 两种方法：
   - **使用 LangChain 的 tool-calling**：将 sub-agent 包装为 `@tool`，作为普通工具传入 supervisor agent
   - **使用 langgraph-supervisor 包**：使用 `create_supervisor` 函数直接传入多个 Agent

3. 使用 `@tool` 装饰器创建一个工具函数，在该函数内部调用 sub-agent 的 `invoke` 方法，并返回 sub-agent 的最终输出（如 `result["messages"][-1].content`）。在 `@tool` 中指定工具名称和描述。

4. `create_supervisor` 来自 `langgraph-supervisor` 包。基本用法：`create_supervisor([agent1, agent2, ...], model=llm, prompt="...")`，返回一个编译好的图。

5. 适用场景：（1）单个 Agent 工具太多，tool-calling 准确率下降；（2）上下文或记忆太大，单个 Agent 无法有效跟踪；（3）任务需要专业化分工（如规划师、研究员、数学专家）。

6. 作者持谨慎态度，认为多智能体系统在当前阶段可能会降低准确性，建议"能不用就不用"。理由是多次尝试后发现结果不稳定，即使调整了 system prompt 也可能产生错误结果。

### 进阶题参考答案

1. System Prompt 的设计直接影响 Supervisor 的任务拆解和工具分配。例如在"38462 + 378 / 49 * 83723 - 123"这类复合运算中，如果 prompt 没有提示"减可用加负数实现"，Supervisor 可能不知道如何处理减法。Prompt 需要清晰说明工具的能力边界和替代方案。

2. **tool-calling 方式**：上下文完全由开发者手动控制，灵活但易出错；错误传播更直接；扩展时需要手动编写新工具包装。**langgraph-supervisor**：自动化程度高，内置上下文管理和 Agent 间通信；错误处理更规范；扩展只需添加 Agent 列表即可。

3. **上下文灾难**：多个 Agent 之间传递信息时，上下文不断膨胀和失真，导致模型在处理关键信息时被噪声淹没。缓解方法：（1）限制每个 Agent 的上下文范围；（2）使用摘要压缩历史；（3）采用 Supervisor 模式集中管理信息流；（4）使用子图隔离不同 Agent 的上下文。

### 编程题参考实现

```python
@tool
def get_order(order_id: str) -> str:
    """Get order status by order ID."""
    orders = {"ORD001": "已发货", "ORD002": "配送中"}
    return orders.get(order_id, "订单不存在")

@tool
def process_refund(order_id: str) -> str:
    """Process refund for an order."""
    return f"订单 {order_id} 退款已处理"

order_agent = create_agent(
    model=llm, tools=[get_order], name="order_agent"
)
refund_agent = create_agent(
    model=llm, tools=[process_refund], name="refund_agent"
)

supervisor_graph = create_supervisor(
    [order_agent, refund_agent],
    model=llm,
    prompt="你是客服主管，根据用户问题路由到订单Agent或退款Agent。"
)
app = supervisor_graph.compile()
```

### 大厂面试题参考解答

1. **多智能体编排模式**：Supervisor（主管分派）、Pipeline（流水线接力）、Swarm（对等协作）、Handoff（智能交接）。Supervisor 模式适合需要统一调度和任务分解的场景，如客服系统、复杂工作流编排。

2. **多 Agent 客服系统设计要点**：（1）分层架构：路由 Agent → 专业 Agent → 升级 Agent；（2）Human-in-the-loop：置信度低时转人工；（3）上下文隔离：每个 Agent 只关注自己的上下文；（4）错误处理：超时、重试、降级机制。

3. **多 vs 单 Agent**：优点是专业化分工、上下文隔离、可扩展性强；缺点是系统复杂度高、通信开销大、错误链更长。建议在以下情况选择多 Agent：单个 Agent 的工具超过 10 个、需要不同领域的专业知识、任务可明确拆分为独立子任务。

4. **LangGraph Supervisor 的任务分发**：Supervisor Agent 通过 LLM 的推理能力分析用户请求，决定调用哪个 sub-agent（当作工具调用）。Sub-agent 执行后返回结果，Supervisor 将结果追加到消息列表，再根据是否需要继续分发或直接回答做出决策。
