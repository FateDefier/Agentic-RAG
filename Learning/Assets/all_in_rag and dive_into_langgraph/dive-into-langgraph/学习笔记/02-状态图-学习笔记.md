## 要点整理

### 一、StateGraph 核心概念

| 概念 | 说明 | 类比 |
|------|------|------|
| **State（状态）** | 节点间共享的数据容器 | 全局变量 / 黑板 |
| **Node（节点）** | 工作流中的处理单元 | 函数 / 步骤 |
| **Edge（边）** | 节点间的连接关系 | 函数调用 / 流程线 |
| **Conditional Edge（条件边）** | 根据状态决定流向的边 | if-else 分支 |
| **START** | 图的入口点 | main 函数 |
| **END** | 图的终止点 | return |

### 二、构建 StateGraph 的步骤

```
创建 StateGraph 实例 → 添加节点 → 添加边/条件边 → 编译 → 执行
```

#### 1. 创建实例
```python
builder = StateGraph(MessagesState)
```

#### 2. 添加节点
```python
builder.add_node('assistant', assistant_func)
builder.add_node('tool', tool_node)
```
- 节点函数接收 `state` 和 `config` 作为参数
- 返回一个字典，包含要更新的 State 字段

#### 3. 添加边
```python
builder.add_edge(START, 'assistant')       # 普通边
builder.add_edge('tool', 'assistant')      # 形成循环
```

#### 4. 添加条件边
```python
builder.add_conditional_edges(
    'assistant',
    should_continue,       # 路由函数
    {'continue': 'tool', 'end': END}  # 路由映射
)
```

#### 5. 编译
```python
my_graph = builder.compile(name='my-graph')
```

### 三、有向图类型

| 类型 | 特点 | 示例 |
|------|------|------|
| DAG（有向无环图） | 无环路，线性执行 | 数据处理流水线 |
| DCG（有向循环图） | 有环路，可循环执行 | ReAct Agent 循环 |

### 四、关键组件详解

- **`MessagesState`**：LangGraph 内置的 State 类型，包含 `messages` 字段
- **`ToolNode`**：预构建的工具执行节点，自动处理工具调用的分发和执行
- **`bind_tools`**：将工具 Schema 绑定到 LLM，使其理解如何调用工具
- **条件路由**：通过检查 `state['messages'][-1].tool_calls` 决定下一步走向

## 参考答案汇总

### 基础题答案

1. **StateGraph 核心三要素：**
   - **节点（Node）**：工作流中的处理单元，每个节点负责一个具体的功能（如 LLM 调用、工具执行）
   - **边（Edge）**：定义节点间的连接关系和执行顺序
   - **状态（State）**：节点间共享的数据容器，所有节点读取和写入同一 State 对象

2. **`add_conditional_edges` vs `add_edge`：**
   - `add_edge`：定义确定的、无分支的流向，A 完成后总是去 B
   - `add_conditional_edges`：定义基于条件的动态路由，根据 State 中的内容决定去哪个节点

3. **条件边**：根据当前 State 动态决定下一步执行路径的边。定义方式：
   ```python
   def router(state: MessagesState, config: RunnableConfig) -> str:
       # 返回一个字符串键，映射到目标节点
       return 'continue' 或 'end'
   ```

4. **有向图类型**：
   - 有向无环图（DAG）和有向循环图（DCG）
   - 天气工作流属于**有向循环图（DCG）**，因为助手节点和工具节点之间存在循环：assistant → tool → assistant

5. **`MessagesState`**：是 LangGraph 预定义的 State 类型，继承自 `TypedDict`，包含一个 `messages` 字段（消息列表）。它是 StateGraph 默认使用的 State 类型，用于存储和管理对话历史。

6. **`ToolNode` 的作用**：是一个预构建节点，负责接收 LLM 输出的工具调用请求，执行对应的工具函数，并将结果以 ToolMessage 形式返回给 State。与直接在助手节点中调用工具相比，ToolNode 实现了关注点分离——助手节点只负责推理和决策，工具节点只负责执行。

### 进阶题参考答案

1. **ReAct Agent vs StateGraph：**
   - `create_agent` 优势：使用简单，自动处理循环和工具分发；劣势：控制粒度粗，难以定制流程
   - StateGraph 优势：完全控制流程，支持条件分支、循环、人机交互等复杂逻辑；劣势：代码量更大，需要手动管理所有细节
   - **选择 StateGraph 的场景**：需要多分支路由（如客服分流）、需要人机交互审批、需要多个工具节点协同、需要精细的状态管理

2. **多分支条件路由设计：**
   ```python
   def router(state: MessagesState, config: RunnableConfig) -> str:
       last_msg = state['messages'][-1]
       content = last_msg.content.lower()
       if '账单' in content:
           return 'billing'
       elif '技术' in content or '报错' in content:
           return 'tech'
       else:
           return 'other'
   
   builder.add_conditional_edges(
       'classifier', router,
       {'billing': 'billing_handler', 'tech': 'tech_handler', 'other': 'escalator'}
   )
   ```

3. **State 的传递和更新机制：** 每个节点函数接收当前 State 作为输入，返回一个 Partial State（部分状态更新）。LangGraph 使用 Reducer 函数合并旧状态和新状态。对于 `messages` 字段，默认使用 `add_messages` reducer，它会将新消息追加到消息列表中而非覆盖。这种机制使得多个节点可以安全地同时更新 State 的不同部分。

### 编程题参考实现

```python
from langgraph.graph import StateGraph, MessagesState, START, END
from langchain_openai import ChatOpenAI
from langchain_core.runnables import RunnableConfig
from langchain.messages import SystemMessage, HumanMessage
from langgraph.types import Command
import os
from dotenv import load_dotenv

_ = load_dotenv()

llm = ChatOpenAI(
    model="qwen3-coder-plus",
    api_key=os.getenv("DASHSCOPE_API_KEY"),
    base_url=os.getenv("DASHSCOPE_BASE_URL"),
)

# 1. 定义节点函数
def classifier(state: MessagesState, config: RunnableConfig):
    """分类节点：判断工单类型"""
    system_prompt = "你是一个工单分类器。请判断用户问题属于: billing（账单）、tech（技术）还是 other（其他）。只返回分类名称。"
    messages = [SystemMessage(system_prompt)] + state['messages']
    response = llm.invoke(messages)
    return {"messages": [response], "category": response.content.strip().lower()}

def handler(state: MessagesState, config: RunnableConfig):
    """处理节点：自动处理工单"""
    category = state.get("category", "other")
    system_prompt = f"你是一个{category}类型的客服助手，请帮助用户解决问题。"
    messages = [SystemMessage(system_prompt)] + state['messages']
    response = llm.invoke(messages)
    return {"messages": [response]}

def escalator(state: MessagesState, config: RunnableConfig):
    """升级节点：转人工处理"""
    response = llm.invoke([
        SystemMessage("该工单需要转接人工客服。请礼貌告知用户正在转接，并说明预计等待时间。"),
        *state['messages']
    ])
    return {"messages": [response]}

# 2. 定义条件路由函数
def route_by_category(state: MessagesState, config: RunnableConfig) -> str:
    category = state.get("category", "other")
    if category in ["billing", "tech"]:
        return "handle"
    return "escalate"

# 3. 构建图
builder = StateGraph(MessagesState)
builder.add_node("classifier", classifier)
builder.add_node("handler", handler)
builder.add_node("escalator", escalator)

builder.add_edge(START, "classifier")
builder.add_conditional_edges(
    "classifier",
    route_by_category,
    {"handle": "handler", "escalate": "escalator"}
)
builder.add_edge("handler", END)
builder.add_edge("escalator", END)

# 4. 编译并测试
graph = builder.compile(name="ticket-system")
```

### 大厂面试题参考解答

1. **【百度】StateGraph 的收益与代价：**
   - **收益**：显式状态管理使流程可追踪、可恢复（checkpoint）；多分支和人机确认场景下比 while 循环+if-else 好维护
   - **代价**：团队需要接受「图」的心智模型；小改动有时要动边和 reducer；样板代码较多
   - **不需要图的业务**：简单的线性 LLM 调用、单轮问答、无状态 API 代理

2. **【大厂通用】Node-Edge-State 关系：** Node 读取 State 并返回更新，Edge 决定 Node 的执行顺序。与传统工作流引擎相比，LangGraph 的独特优势在于：
   - State 是**隐式共享**的（节点间无需显式传递参数）
   - 支持**循环图**（DAG 框架不支持循环）
   - 内置 LLM 调用和工具执行的原生支持

3. **【腾讯/字节】避免无限循环：** 在条件路由函数中设置最大循环次数检查（如 `len(state['messages']) > MAX_TURNS` 时强制结束），或者在 State 中添加 `turn_count` 计数器，每次循环递增并在达到阈值时返回 END。

4. **【阿里】编译过程：** `compile()` 方法将 StateGraph 转换为 CompiledStateGraph，此过程中会：
   - 验证图的拓扑结构（是否有不可达节点、是否有未连接节点）
   - 优化执行计划（确定节点的执行顺序和并行可能性）
   - 注入 checkpoint 和中间件支持
   - 生成可序列化的运行时图
   - CompiledStateGraph 是可直接调用的 Runnable，而 StateGraph 是图的定义/配置阶段
