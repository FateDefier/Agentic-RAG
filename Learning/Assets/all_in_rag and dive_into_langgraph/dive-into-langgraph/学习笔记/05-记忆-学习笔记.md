## 要点整理

### 一、记忆体系总览

```
LangGraph 记忆体系
├── 短期记忆（MemorySaver / Checkpointer）
│   ├── 作用：临时存储 Agent 工作状态
│   ├── 存储内容：消息历史、节点执行位置
│   ├── 生命周期：随线程（thread）存在
│   └── 实现：InMemorySaver / SqliteSaver / PostgresSaver 等
├── 长期记忆（MemoryStore）
│   ├── 作用：持久化保存业务信息
│   ├── 存储内容：用户偏好、配置参数、业务数据
│   ├── 生命周期：跨线程、跨会话持久存在
│   └── 实现：InMemoryStore（可替换为外部存储）
└── LangMem（第三方库）
    ├── 提供更高级的记忆存取功能
    └── 可与其他记忆模块配合使用
```

### 二、短期记忆（Checkpointer）

| 检查点方案 | 存储位置 | 持久化 | 适用场景 |
|-----------|----------|--------|----------|
| `InMemorySaver` | 内存 | ❌ | 开发测试、演示 |
| `SqliteSaver` | SQLite 文件 | ✅ | 单机部署、小型应用 |
| `PostgresSaver` | PostgreSQL | ✅ | 企业级、分布式 |
| `MongoDBSaver` | MongoDB | ✅ | 文档型需求 |
| `RedisSaver` | Redis | 需配置 | 高频读写、缓存场景 |

**核心作用**：
- 跨对话轮次保持上下文
- 支持中断恢复（配合 HITL）
- 支持时间旅行（Time Travel）

### 三、长期记忆（MemoryStore）

```python
# 创建带语义搜索的存储
store = InMemoryStore(index={"embed": embed_func, "dims": 1024})

# 写入记忆
store.put(("users",), "user_1", {"name": "John", "language": "English"})

# 直接读取
item = store.get(("users",), "user_1")

# 语义搜索
items = store.search(("users",), query="language preferences")
```

| 操作 | 方法 | 说明 |
|------|------|------|
| 写入 | `store.put(namespace, key, value)` | 存入数据 |
| 读取 | `store.get(namespace, key)` | 通过 key 精确读取 |
| 搜索 | `store.search(namespace, query)` | 语义相似度搜索 |
| 过滤 | `store.search(namespace, query, filter=...)` | 带条件过滤的搜索 |

### 四、工具访问长期记忆

## 参考答案汇总

### 基础题答案

1. **记忆模块不是必需的**，因为 StateGraph 自带的 `messages` 列表已满足基础"记忆"需求。需要添加 Memory 模块的三种情况：
   - 历史消息太多，需使用外部工具存储
   - 触发人工干预（interrupt），需临时保存 Agent 状态
   - 跨对话提取用户偏好等需要长期记忆的场景

2. **两种记忆类型：**
   - **短期记忆（MemorySaver/Checkpointer）**：临时存储 Agent 或工作流的状态，用于失败恢复和跨轮次对话保持上下文
   - **长期记忆（MemoryStore）**：持久化保存业务相关信息（用户属性、偏好等），支持语义检索

3. **短期记忆通过 Checkpointer 检查点机制实现。** `InMemorySaver` 将状态存储在内存中，程序重启后丢失；`SqliteSaver` 将状态持久化到 SQLite 数据库文件，程序重启后仍可恢复。

4. **`checkpointer` 参数传递给 `builder.compile(checkpointer=...)` 方法。** 如果不提供 checkpointer，每次调用工作流时都是从空白状态开始，不会保留历史对话信息。

5. **`InMemoryStore` 支持语义搜索需要配置 `index` 参数**，包含：
   - `embed`：嵌入（Embedding）生成函数
   - `dims`：嵌入向量的维度
   配置后可以通过 `store.search(namespace, query="...")` 进行语义检索。

6. **`LangMem`** 是一个独立的第三方库（由 LangChain 团队开发），提供更高级的记忆存取功能。它可以与 LangGraph 内置的 MemorySaver/MemoryStore 配合使用，也可以独立使用。LangMem 提供了更丰富的记忆管理 API。

### 进阶题参考答案

1. **检查点方案选型：**
   - `InMemorySaver`：仅用于开发和测试，数据不持久化
   - `SqliteSaver`：适合单机或小团队项目，零配置，足够可靠
   - `PostgresSaver`：适合企业级生产环境，支持分布式部署、事务完整性、高并发

2. **Namespace 设计：** 在多租户系统中，Namespace 应包含租户 ID 作为前缀，如 `("tenant_a", "users")` 和 `("tenant_b", "users")`。这样不同租户的数据在逻辑上完全隔离，搜索和读取时也在各自 Namespace 范围内进行。还可以按业务域进一步细分：`("tenant_a", "users", "preferences")`。

3. **数据流路径：** 用户在消息中表达偏好 → Agent 调用 `save_preference` 工具 → `@tool` 函数通过 `runtime.store` 将数据写入 `InMemoryStore` → 后续对话中 Agent 调用 `get_preference` 工具 → 通过 `runtime.store.get()` 读取长期记忆 → Agent 基于记忆个性化回复用户。

### 编程题参考实现

```python
import os
from dotenv import load_dotenv
from dataclasses import dataclass
from typing_extensions import TypedDict
from openai import OpenAI
from langchain_openai import ChatOpenAI
from langchain.agents import create_agent
from langchain.tools import tool, ToolRuntime
from langgraph.store.memory import InMemoryStore

_ = load_dotenv()

# 1. 配置 Embedding 和模型
EMBED_DIM = 1024
client = OpenAI(
    api_key=os.getenv("DASHSCOPE_API_KEY"),
    base_url=os.getenv("DASHSCOPE_BASE_URL"),
)

def embed(texts: list[str]) -> list[list[float]]:
    response = client.embeddings.create(
        model="text-embedding-v4",
        input=texts,
        dimensions=EMBED_DIM,
    )
    return [item.embedding for item in response.data]

llm = ChatOpenAI(
    model="qwen3-coder-plus",
    api_key=os.getenv("DASHSCOPE_API_KEY"),
    base_url=os.getenv("DASHSCOPE_BASE_URL"),
)

# 2. 创建 InMemoryStore
store = InMemoryStore(index={"embed": embed, "dims": EMBED_DIM})

# 3. 定义 Context 和工具
@dataclass
class Context:
    user_id: str

class UserPreference(TypedDict):
    preference_type: str
    preference_value: str

@tool
def save_preference(preference: UserPreference, runtime: ToolRuntime[Context]) -> str:
    """保存用户偏好，如语言偏好、主题偏好等"""
    user_id = runtime.context.user_id
    prefs = runtime.store.get(("preferences",), user_id)
    existing = prefs.value if prefs else []
    existing.append(preference)
    runtime.store.put(("preferences",), user_id, existing)
    return f"已保存偏好: {preference['preference_type']} = {preference['preference_value']}"

@tool
def get_preference(runtime: ToolRuntime[Context]) -> str:
    """查询用户的所有偏好"""
    user_id = runtime.context.user_id
    prefs = runtime.store.get(("preferences",), user_id)
    if prefs:
        return str(prefs.value)
    return "暂无偏好记录"

# 4. 创建 Agent
agent = create_agent(
    model=llm,
    tools=[save_preference, get_preference],
    store=store,
    context_schema=Context,
)

# 5. 测试保存偏好
agent.invoke(
    {"messages": [{"role": "user", "content": "我喜欢用简洁的语言回复"}]},
    context=Context(user_id="user_123"),
)

# 6. 测试查询偏好
result = agent.invoke(
    {"messages": [{"role": "user", "content": "我的偏好是什么？"}]},
    context=Context(user_id="user_123"),
)
for msg in result['messages']:
    msg.pretty_print()
```

### 大厂面试题参考解答

1. **【百度】短期 vs 长期记忆：**
   - 短期记忆通过 `checkpointer` 参数在 `compile()` 时注入，本质上是在每个步骤后保存 State 快照到检查点
   - 长期记忆通过 `store` 参数在 `create_agent()` 时注入，通过工具函数中的 `runtime.store` 进行读写
   - 项目中一般两种都用：短期记忆保持对话上下文，长期记忆存储用户画像

2. **【大厂通用】Agent 记忆 vs 传统对话管理：**
   - 传统系统（Rasa/Dialogflow）的记忆通常是 slot-filling 模式，记忆结构固定
   - LangGraph 的 Checkpoint 保存的是**完整状态快照**，可以精确恢复到任意历史节点
   - Checkpoint 带来的额外能力：中断恢复、时间旅行、分支执行、A/B 测试

3. **【字节跳动】记忆长度 vs Token 消耗平衡策略：**
   - 消息截断（只保留最近的 N 条消息）
   - 对话摘要（SummarizationMiddleware 自动压缩）
   - 滑动窗口（保留首条 + 最近消息）
   - 分层记忆（短期上下文窗口 + 长期向量存储检索）

4. **【阿里】LangGraph 长期记忆 vs 外部数据库：**
   - LangGraph 的 `InMemoryStore` 优点：与框架深度集成，支持语义搜索，无需额外基础设施
   - 外部数据库（如 Redis/PostgreSQL）优点：成熟稳定，监控完善，团队熟悉度高
   - Trade-off：`InMemoryStore` 简化了开发但可能在超大规模下有限制；外部数据库灵活但需要更多维护工作。建议中小项目先用 `InMemoryStore`，必要时切换到外部存储实现
