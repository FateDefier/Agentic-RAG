## 大厂面试题精选

1. **LangGraph 中如何实现并行执行？有哪些方式？**（AI 工程化岗）
   - 来源：LangGraph 面试高频题
   - 解析：节点并发（并联节点）、`@task` 装饰器（声明并发单元）、Map-Reduce（Send 函数分发）、Sub-graphs（子图并联）。

2. **Map-Reduce 模式在 LangGraph 中是如何实现的？`Send` 函数的作用是什么？**（系统设计岗）
   - 来源：慕课网/Stack Overflow 相关面试题
   - 解析：Send 函数动态创建并行分支，将参数分发给指定节点处理，reduce 节点通过 reducer 合并结果。

3. **进程、线程、协程的区别是什么？LangGraph 为何主要使用协程级别的并发？**（基础算法岗）
   - 来源：多家大厂基础题
   - 解析：进程资源隔离但切换成本高，线程共享内存需处理锁，协程用户态切换开销最低。LangGraph 基于 asyncio 实现协程并发，适合 I/O 密集型场景。

4. **LangGraph 中节点默认对相同 key 是"覆盖写"的，如何实现累加/合并？**（LangGraph 专项题）
   - 来源：ModelEngine 社区面试题
   - 解析：需要为 key 指定 reducer（如 `Annotated[list, operator.add]` 或 `add_messages`），工具返回的 Partial 会按 reducer 聚合。

5. **设计一个需要并行处理的 Agent 工作流：比如同时对多个文档进行摘要，然后合并结果。你会选择哪种并发方式？**（AI 应用设计岗）
   - 来源：Agent Engineer 面试题
   - 解析：Map-Reduce 模式最合适——Send 分发文档到多个 worker 节点并行摘要，reduce 节点通过 reducer 合并结果。

## 要点整理

### 四种并发实现方式

| 方式 | 实现原理 | 适用场景 | 关键要素 |
|------|---------|---------|---------|
| **节点并发** | 并联多个节点，从 START 同时出发 | 独立的多个处理步骤 | `add_edge(START, node_a)` 到多个节点 |
| **`@task` 装饰器** | 将函数声明为并发执行单元，返回 future | I/O 密集型并行任务 | `@task` + `@entrypoint` + `f.result()` |
| **Map-Reduce** | `Send` 函数分发参数到并行节点 | 批量数据处理 | `Send("node", params)` + reducer |
| **Sub-graphs** | 并联多个子图作为父图节点 | 复用已有工作流 | `add_node("sub", compiled_subgraph)` |

### 关键概念

| 概念 | 说明 |
|------|------|
| **并行（Parallelism）** | 同一时刻多个任务真正同时执行（多核 CPU） |
| **并发（Concurrency）** | 同一时间段多个任务交替执行（单核时间片轮转） |
| **GIL** | 全局解释器锁，限制 Python 多线程并行 |
| **Send** | LangGraph 的并行分发函数，动态创建分支 |
| **Reducer** | 用于合并多个节点写入同一 key 的值 |
| **Sub-graph** | 可嵌套在父图中的已编译图，支持 `xray=1` 查看内部 |

### 进程/线程/协程对比

| 维度 | 进程 | 线程 | 协程 |
|------|------|------|------|
| 资源占用 | 高（独立内存空间） | 中（共享内存） | 低（用户态） |
| 切换成本 | 高（内核态） | 中（内核态） | 极低（用户态） |
| 调度方式 | 操作系统 | 操作系统 | 程序自己 |
| 适用场景 | CPU 密集型 | I/O 密集型 | I/O 密集型 |
| 类比 | 一栋独立房子 | 房子里的工人 | 工人主动切换任务 |

## 参考答案汇总

### 基础题答案

1. 由于 GIL（全局解释器锁）的存在，LangGraph 主要实现的是**并发**而非真正的多核并行。标题使用"并行"一词是为了与官方文档保持一致，因为官方文档大量使用 parallelization 这个词。

2. 四种方式：
   - **节点并发**：通过并联节点实现，多个节点从 START 同时出发
   - **`@task` 装饰器**：将函数声明为并发执行单元，配合 `@entrypoint` 使用
   - **Map-Reduce**：用 `Send` 函数并发地发送参数给节点执行，然后聚合结果
   - **Sub-graphs**：通过并联子图实现并发，可复用已有工作流

3. **节点并发**：在 StateGraph 中将多条边从 START 节点指向不同节点，这些节点会同时执行。例如 `builder.add_edge(START, 'node_a')` 和 `builder.add_edge(START, 'node_b')`，node_a 和 node_b 会并发执行。

4. `@task` 装饰器将函数声明为并发执行单元，调用该函数会返回 future 对象。`@entrypoint` 装饰器定义工作流入口函数，可在其中并发调用多个 `@task` 函数并收集 future 结果。

5. Map-Reduce 模式通过 `Send` 函数实现：`Send` 将参数分发给指定节点并发执行（Map 阶段），然后通过 reducer（如 `operator.add`）将所有分支的结果合并（Reduce 阶段）。`Send` 函数用于动态创建并行分支。

6. Sub-graphs 是将已经编译好的图作为节点添加到父图中。当父图中的多个子图节点都从 START 出发时，这些子图会并发执行。子图内部可以有自己的状态和逻辑。

### 进阶题参考答案

1. **进程**是资源分配的基本单位，有独立内存空间；**线程**是进程内的执行流，共享内存；**协程**是用户态轻量级线程，协作式调度。LangGraph 的并发实现主要基于**协程**层（asyncio），因为协程切换成本最低，适合 Agent 工作流中大量的 I/O 等待（如 LLM 调用、工具调用）。

2. Map 节点的 state schema 可以不同于全局 schema，因为每个 Map 节点处理的是子任务，有自己的独立状态。这种设计的好处是：（1）每个分支只需关注自己的输入数据，schema 更简洁；（2）避免全局 schema 过度复杂；（3）提高模块化和可复用性。

3. 作者认为 LangGraph 的并发机制在某些场景下写复杂了，使用 `concurrent.futures.ThreadPoolExecutor` 更简单直接。两种方式各有优劣：LangGraph 方式更标准化、与图结构集成更好；`ThreadPoolExecutor` 更简单灵活，但不具备 LangGraph 的可视化和持久化特性。

### 编程题参考实现

```python
@task
def generate_paragraph(topic: str) -> str:
    response = llm.invoke([
        {"role": "system", "content": "你是一个会写科普文的智能助手"},
        {"role": "user", "content": f"写一个关于{topic}的段落，大约100字。"}
    ])
    return response.content

@entrypoint(checkpointer=InMemorySaver())
def workflow(topics: list[str]) -> str:
    futures = [generate_paragraph(topic) for topic in topics]
    paragraphs = [f.result() for f in futures]
    return "\n\n".join(paragraphs)

result = workflow.invoke(["AI 发展史", "机器学习基础", "深度学习应用"])
```

### 大厂面试题参考解答

1. **LangGraph 并行方式**：四种方式各有侧重。节点并发适合固定数量并行任务，`@task` 适合函数级并发，Map-Reduce 适合大规模数据并行处理，Sub-graphs 适合工作流复用。选择依据是任务的静态/动态特性、数据规模、复用需求。

2. **Send 函数**：LangGraph 的 `Send` 函数是条件边的特殊形式，它返回一个 Send 对象列表，每个 Send 对象包含目标节点名称和独立的状态参数。LangGraph 运行时接收到 Send 列表后，会为每个 Send 创建独立的执行分支，这些分支并发执行。

3. **协程优势**：Agent 工作流中大部分时间是等待 LLM 响应或工具调用返回，属于 I/O 密集场景。协程在等待时主动让出执行权，让其他协程运行，资源利用率高。线程需要操作系统调度，切换开销大；进程更重。

4. **Reducer 机制**：在定义 State 时使用 `Annotated[类型, reducer函数]`，如 `Annotated[list, operator.add]`。当多个节点返回包含相同 key 的 dict 时，reducer 会决定如何合并这些值（如 list 追加、数值求和）。

5. **文档并行摘要设计**：选择 Map-Reduce 模式。Map 阶段：使用 `Send` 将每个文档分发给 `summarize_node` 并发处理。Reduce 阶段：使用带 `Annotated[list, operator.add]` reducer 的 key 收集所有摘要。最终由一个合并节点进行最终整理。
