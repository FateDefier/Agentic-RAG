## 要点整理

### 一、中间件概述

- **中间件 = 钩子函数（Hook）**：在工作流的关键节点预埋可扩展逻辑
- **核心优势**：在不修改 Agent 核心逻辑的前提下增强功能
- **应用场景**：预算控制、消息截断、敏感词过滤、PII 检测、动态提示词、上下文注入等

### 二、装饰器类型速查表

| 装饰器 | 执行时机 | 典型用途 |
|--------|----------|----------|
| `@before_agent` | Agent 执行前 | 敏感词过滤、权限检查、请求预处理 |
| `@after_agent` | Agent 执行后 | 结果后处理、日志记录、统计分析 |
| `@before_model` | 每次模型调用前 | 消息截断、上下文修剪 |
| `@after_model` | 模型收到响应后 | 响应过滤、格式转换 |
| `@wrap_model_call` | 控制模型调用过程 | 动态切换模型、修改请求参数 |
| `@wrap_tool_call` | 控制工具调用过程 | 工具调用审计、频率限制 |
| `@dynamic_prompt` | 动态生成系统提示词 | 注入用户信息、时间、业务上下文 |
| `@hook_config` | 配置钩子行为 | 全局中间件配置 |

### 三、四大实战案例

| 案例 | 使用的中间件 | 核心逻辑 | 效果 |
|------|-------------|----------|------|
| **预算控制** | `@wrap_model_call` | 根据消息数动态切换模型 | 对话超5条切到低费率模型 |
| **消息截断** | `@before_model` | 保留首条+最近3条消息 | 控制上下文长度 |
| **敏感词过滤** | `@before_agent` + `can_jump_to` | 检测敏感词，拒绝请求 | 提前阻断不安全请求 |
| **PII 检测** | `@before_agent` | 用可信模型检测PII后处理 | 拒绝/屏蔽隐私信息 |

### 四、中间件编写范式

```python
@decorator_type(can_jump_to=["end"])  # 可选参数
def my_middleware(state: AgentState, runtime: Runtime) -> dict[str, Any] | None:
    # 1. 读取 state 中的信息
    # 2. 执行自定义逻辑
    # 3. 返回 None（不做修改）或返回 dict（修改 state）
    # 4. 通过 "jump_to" 控制流程跳转
```

### 五、内置中间件类

LangChain 还提供了多种内置中间件类，如 `SummarizationMiddleware`（自动摘要）、`ModelCallLimitMiddleware`（模型调用次数限制）、`ToolCallLimitMiddleware`（工具调用限制）、`ModelFallbackMiddleware`（模型回退）等。

## 参考答案汇总

### 基础题答案

1. **中间件定义：** 中间件（Middleware）是一种钩子函数（Hook），预埋在 Agent 工作流的特定位置。当工作流执行到该位置时，会自动触发中间件中的逻辑。中间件可以读取/修改 State、控制流程跳转、注入额外逻辑，而无需修改 Agent 的核心代码。

2. **8 种装饰器类型：**

   | 装饰器 | 作用 |
   |--------|------|
   | `@before_agent` | 在 Agent 执行前执行逻辑 |
   | `@after_agent` | 在 Agent 执行后执行逻辑 |
   | `@before_model` | 在每次模型调用前执行逻辑 |
   | `@after_model` | 在每次模型收到响应后执行逻辑 |
   | `@wrap_model_call` | 控制模型的调用过程 |
   | `@wrap_tool_call` | 控制工具的调用过程 |
   | `@dynamic_prompt` | 动态生成系统提示词 |
   | `@hook_config` | 配置钩子行为 |

3. **`@before_model` vs `@wrap_model_call`：**
   - `@before_model`：在模型调用前执行，适合读取/修改 state（如截断消息），但不能干涉模型调用本身
   - `@wrap_model_call`：包装模型调用过程，可以完全控制「调用哪个模型」「如何调用」「如何处理响应」
   - 选择：只需在模型前调整数据用 `@before_model`；需要动态切换模型或修改调用参数用 `@wrap_model_call`

4. **护栏（Guardrails）**：是 Agent 提供的内容安全能力统称。大模型自身有风控能力但易被绕过，中间件可以在工程层面进行强制检查。LangGraph 中通过 `@before_agent(can_jump_to=["end"])` 在 Agent 执行前检查输入，发现违规直接跳转到 END 终止执行。

5. **`can_jump_to=["end"]` 的作用：** 允许中间件通过返回 `{"jump_to": "end"}` 来跳过 Agent 的正常执行流程，直接跳转到工作流的 END 节点，从而阻断不安全请求。

6. **PII 检测**：识别用户输入中的个人可识别信息（邮箱、IP、地址、银行卡、用户名等）。中间件处理方式有两种：
   - **拒绝回答**：检测到 PII 后直接终止请求
   - **屏蔽信息**：用 `*****` 替换敏感信息后再交给 Agent 处理

### 进阶题参考答案

1. **多级预算控制方案：**
   ```python
   @wrap_model_call
   def multi_level_model_selection(request: ModelRequest, handler) -> ModelResponse:
       msg_count = len(request.state["messages"])
       if msg_count <= 5:
           model = gpt4_model
       elif msg_count <= 10:
           model = gpt35_model
       else:
           model = local_model
       return handler(request.override(model=model))
   ```

2. **中间件执行顺序：**
   - `@before_agent`（最外层，Agent 开始前）→ `@before_model`（每次模型调用前）→ 模型调用 → `@wrap_model_call` 的 handler 部分 → `@after_model` → `@after_agent`
   - 注意 `@wrap_model_call` 是一个包装器，它在模型调用前和调用后都有执行机会

3. **「拒绝回复」vs「屏蔽隐私信息」：**
   - **拒绝回复**：适用于对隐私要求极高的场景（如金融、医疗），但用户体验差
   - **屏蔽信息**：适用于一般隐私保护场景，用户可继续得到有用回答，但可能因屏蔽过度影响上下文
   - 折中方案：屏蔽后提示用户哪些信息被屏蔽了

### 编程题参考实现

```python
from langchain.agents.middleware import before_agent, AgentState
from langgraph.runtime import Runtime
from typing import Any
import re
import requests
from langchain_openai import ChatOpenAI
from langchain.agents import create_agent
import os
from dotenv import load_dotenv

_ = load_dotenv()

llm = ChatOpenAI(
    model="qwen3-coder-plus",
    api_key=os.getenv("DASHSCOPE_API_KEY"),
    base_url=os.getenv("DASHSCOPE_BASE_URL"),
)

def extract_urls(text: str) -> list[str]:
    url_pattern = r'https?://[^\s]+'
    return re.findall(url_pattern, text)

@before_agent(can_jump_to=["end"])
def url_preprocessor(state: AgentState, runtime: Runtime) -> dict[str, Any] | None:
    if not state["messages"]:
        return None
    last_msg = state["messages"][-1]
    if last_msg.type != "human":
        return None
    
    content = last_msg.content
    urls = extract_urls(content)
    if not urls:
        return None
    
    summary_model = ChatOpenAI(...)
    processed_content = content
    
    for url in urls:
        try:
            response = requests.get(url, timeout=5)
            summary_prompt = f"请提取以下网页内容的核心信息摘要：\n{response.text[:2000]}"
            summary = summary_model.invoke(summary_prompt).content
            processed_content = processed_content.replace(url, f"[摘要: {summary}]")
        except:
            return {
                "messages": [{"role": "assistant", "content": f"无法访问链接 {url}，请检查链接有效性。"}],
                "jump_to": "end"
            }
    
    return {"messages": [{"role": "user", "content": processed_content}]}
```

### 大厂面试题参考解答

1. **【百度】中间件解决的问题：** 中间件解决了 Agent 在「横切关注点」（cross-cutting concerns）上的代码复用问题。例如：所有 Agent 都需要日志记录、权限检查、内容安全过滤，如果每个 Agent 都重复实现这些逻辑，代码会非常冗余。中间件将这些通用逻辑抽取为可插拔的钩子，业务代码和基础设施逻辑完全解耦。

2. **【大厂通用】护栏架构层次：**
   - **输入层**：检查用户输入是否包含敏感词、PII、恶意指令（`@before_agent`）
   - **模型层**：控制模型调用，防止 Prompt Injection（`@before_model` / `@wrap_model_call`）
   - **工具层**：控制工具调用的参数和频率（`@wrap_tool_call`）
   - **输出层**：过滤模型输出中的不当内容（`@after_model`）
   - LangGraph 中间件可以作用在上述所有层次，提供统一的护栏实现框架

3. **【字节跳动】工具调用频率控制：** 使用 `@wrap_tool_call` 包装工具调用，在包装函数中检查调用频率。可以维护一个 `tool_call_counter` 字典，记录每个工具在时间窗口内的调用次数，超过阈值则拒绝或降级处理。

4. **【阿里】动态上下文注入：** 使用 `@dynamic_prompt` 装饰器最合适。它专门用于在每次 Agent 调用时动态生成系统提示词，可以注入当前时间、用户位置、会话 ID 等信息。核心实现是在装饰器函数中拼接包含动态信息的系统提示词字符串。
