## 大厂面试题精选

1. **什么是上下文工程（Context Engineering）？它与 Prompt Engineering 有什么区别？**（大模型架构岗）
   - 来源：综合多家大厂面试
   - 解析：Context Engineering 是动态管理和组装上下文的技术，比 PE 更关注"让模型看到什么"而非"如何对模型说话"。

2. **LangGraph 中的三种上下文类型是什么？在实际业务中如何应用？**（AI应用开发岗）
   - 来源：LangGraph 生态企业面试题
   - 解析：Model Context（模型上下文）、Tool Context（工具上下文）、Life-cycle Context（生命周期上下文），分别对应 Prompt 修改、工具执行信息、消息历史压缩。

3. **如何解决 Agent 中的"上下文腐烂"（Context Rot）问题？**（高级算法岗）
   - 来源：Chroma 团队相关研究引申出的面试题
   - 解析：上下文过长导致模型性能退化，可通过 SummarizationMiddleware 压缩、选择性检索、结构化笔记等方式解决。

4. **Agent 的多轮对话中，如何保证上下文不丢失且不被噪声污染？**（大模型应用岗）
   - 来源：ByteDance/百度面试
   - 解析：结合 State（短期记忆）+ Store（长期记忆）+ Runtime（运行时环境），利用中间件动态构建纯净上下文。

## 要点整理

### 三大上下文类型

| 上下文类型 | 说明 | 实现方式 |
|-----------|------|---------|
| **模型上下文（Model Context）** | 动态修改 system prompt | `@dynamic_prompt` 中间件 |
| **工具上下文（Tool Context）** | 工具执行时可访问的信息 | `ToolRuntime`、`context_schema` |
| **生命周期上下文（Life-cycle Context）** | 持续更新，旧消息替换为摘要 | `SummarizationMiddleware` |

### 三种存储介质

| 存储介质 | 特点 | 用途 | 访问方式 |
|---------|------|------|---------|
| **Runtime** | 所有节点共享，时效性高 | 用户角色、部署环境 | `request.runtime.context` |
| **State** | 节点间顺序传递 | Prompt、AI Message | `request.state["messages"]` |
| **Store** | 持久化存储，跨 Workflow | 用户偏好、统计值 | `request.runtime.store.get()` |

### 核心中间件

| 中间件 | 功能 | 使用场景 |
|--------|------|---------|
| `@dynamic_prompt` | 动态修改 system prompt | 根据对话轮次、用户偏好等调整提示词 |
| `@wrap_model_call` | 动态修改消息列表 | 注入文件内容、外部数据到消息列表 |
| `SummarizationMiddleware` | 压缩长上下文 | 防止 Context Rot，保持模型性能 |

### Schema 定义方式

- `dataclasses`：轻量级，适合简单结构
- `pydantic`：带验证，适合复杂场景
- `TypedDict`：TypeScript 风格，适合类型提示

## 参考答案汇总

### 基础题答案

1. **三种上下文：**
   - **模型上下文（Model Context）**：与模型调用直接相关的上下文，如 system prompt 的动态修改，通过 `@dynamic_prompt` 中间件实现。
   - **工具上下文（Tool Context）**：工具在执行时可以访问的信息，通过 `ToolRuntime` 和 `context_schema` 注入。
   - **生命周期上下文（Life-cycle Context）**：在整个 Agent 生命周期中持续更新的上下文，如将旧消息替换为摘要，通过 `SummarizationMiddleware` 实现。

2. 可以使用 `dataclasses`、`pydantic`、`TypedDict` 这些包的任意一个创建 Context Schema。

3. `@dynamic_prompt` 中间件用于动态修改系统提示词（system prompt）。它可以获取以下三个存储介质中的变量：
   - **Runtime（运行时）**：所有节点共享，时效性要求高的信息
   - **State（短期记忆）**：节点间顺序传递，存储 Prompt 和 AI Message
   - **Store（长期记忆）**：持久化存储，可跨 Workflow 保存信息

4. `@wrap_model_call` 中间件用于动态修改消息列表（Messages），可以在模型调用前对消息进行增删改。与 `@dynamic_prompt` 的区别：前者修改的是完整的消息列表，后者只修改 system prompt。

5. 生命周期上下文会在 Agent 的整个生命周期中持续更新，将旧消息替换为摘要。`SummarizationMiddleware` 中间件负责维护它。

6. - **Runtime**：所有节点共享一个 Runtime，同一时刻取值相同，适合存储时效性要求高的信息（如用户角色、部署环境）
   - **State**：节点之间按顺序传递，每个节点接收上一个节点处理后的 State，主要用于存储 Prompt 和 AI Message
   - **Store**：负责持久化存储，可以跨 Workflow/Agent 保存信息，适合存用户偏好、历史统计值等

### 进阶题参考答案

1. **Runtime** 适合存储与当前请求绑定的信息（如用户角色、IP、环境标识），因为它所有节点共享且生命周期短。**State** 适合存储对话流转中的中间数据（如消息列表），因为它天然按节点顺序传递。**Store** 适合需要跨会话保留的数据（如用户偏好、知识库），因为它支持持久化。建议：将 Session 级别的数据放 Runtime，对话流数据放 State，跨会话数据放 Store。

2. `SummarizationMiddleware` 的工作原理：当上下文达到触发条件（如 token 数超过阈值）时，自动对旧消息进行摘要压缩，保留最近的 N 条消息。触发值设置建议：`max_tokens_before_summary=3000`，`messages_to_keep=20`。过小会导致频繁压缩丢失信息，过大会导致上下文过长影响模型性能。

3. 可以设计一个包含 `token`、`user_id` 字段的 Context Schema，使用 `@dataclass` 或 Pydantic 定义。在工具函数中通过 `ToolRuntime[Context]` 获取 `runtime.context.token` 等认证信息。每个工具函数的参数中声明 `runtime: ToolRuntime[Context]` 即可自动注入。

### 编程题参考实现

```python
@dataclass
class UserContext:
    user_id: str
    language: str

@dynamic_prompt
def user_aware_prompt(request: ModelRequest) -> str:
    user_id = request.runtime.context.user_id
    store = request.runtime.store
    user_prefs = store.get(("user_preferences",), user_id)
    
    base = "You are a helpful customer service assistant."
    if user_prefs:
        lang = user_prefs.value.get("language", "zh")
        base += f"\nPlease respond in {lang}."
    return base

@tool
def get_user_history(user_id: str, runtime: ToolRuntime) -> str:
    """Get user's history issues."""
    store = runtime.store
    history = store.get(("user_history",), user_id)
    return history.value if history else "No history"

store = InMemoryStore()
agent = create_agent(
    model=llm,
    tools=[get_user_history],
    middleware=[user_aware_prompt],
    context_schema=UserContext,
    store=store,
)
```

### 大厂面试题参考解答

1. **Context Engineering vs Prompt Engineering**：Prompt Engineering 关注如何编写有效的提示词指令给 LLM，而 Context Engineering 关注在正确的时间向 LLM 提供正确的信息和工具。前者是"教模型怎么说"，后者是"给模型看什么"。Context Engineering 是 PE 的上位概念，包含 RAG、记忆管理、工具编排等。

2. **三种上下文的应用**：Model Context 用于个性化 system prompt（如根据用户角色调整回答风格）；Tool Context 用于在工具中访问运行时信息（如数据库连接）；Life-cycle Context 用于长对话管理（自动摘要压缩历史消息）。

3. **Context Rot 解决方案**：（1）使用 SummarizationMiddleware 自动压缩；（2）采用选择性检索而非全量注入；（3）使用结构化笔记方式管理历史；（4）设置合理的 token 预算和触发阈值。

4. **多轮对话上下文管理**：利用 State 保存最近 N 轮对话，利用 Store 持久化用户长期偏好，利用 Runtime 注入当前会话环境变量。通过 `@dynamic_prompt` 和 `@wrap_model_call` 中间件动态组装最终送入模型的上下文。
