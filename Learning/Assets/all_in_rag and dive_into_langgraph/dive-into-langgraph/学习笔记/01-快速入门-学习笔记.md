## 要点整理

### 一、ReAct Agent 基础

- **ReAct = Reasoning + Acting**：将推理与行动结合的智能体范式
- **工作流程三循环**：思考（Think）→ 行动（Act）→ 观察（Observe），循环直到任务完成
- **核心优势**：通过工具调用自动补足未知上下文信息，基于新信息做出下一步决策

### 二、环境配置

| 组件 | 说明 |
|------|------|
| `langgraph` | 核心框架 |
| `langchain-openai` | OpenAI / 兼容接口（如阿里百炼 Qwen） |
| `python-dotenv` | 环境变量管理 |
| `create_agent` | 快速创建 Agent 的工厂函数 |

### 三、Agent 创建方式

| 类型 | 代码 | 说明 |
|------|------|------|
| 简单 Agent | `create_agent(model=llm, system_prompt=...)` | 无工具调用 |
| 带工具 Agent | `create_agent(model=llm, tools=[...])` | 自动决定是否调用工具 |
| 带权限 Agent | `create_agent(...)` + `ToolRuntime` + `context` | 在运行时注入上下文控制权限 |

### 四、结构化输出

- 使用 Pydantic 的 `BaseModel` 定义输出格式
- 在 `create_agent` 中设置 `response_format=YourModel`
- 通过 `response['structured_response']` 获取结构化的返回结果

### 五、流式输出

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| `updates` | 每个节点完成后更新 | 查看各节点的执行结果 |
| `messages` | 每个 token 生成后更新 | 实时显示生成内容 |
| `values` | 流式输出 State 快照 | 查看工具调用信息 |

## 参考答案汇总

### 基础题答案

1. **ReAct Agent 的工作流程遵循「思考→行动→观察」三步循环：**
   - **思考（Reasoning）**：LLM 分析当前输入和历史信息，决定下一步需要做什么
   - **行动（Acting）**：执行一个具体的动作，可能是调用工具、查询数据库或直接回应
   - **观察（Observing）**：获取行动的结果反馈，将新信息加入上下文
   - 循环持续进行，直到 LLM 判断任务完成或无法继续

2. **`create_agent` 的核心参数：**
   - `model`：LLM 模型实例
   - `tools`：工具函数列表（可选）
   - `system_prompt`：系统提示词
   - `response_format`：结构化输出格式（可选）
   - `middleware`：中间件列表（可选）
   - `checkpointer`：检查点/记忆存储（可选）

3. **结构化输出**：指让 Agent 返回特定格式（如 JSON Schema）的数据而非纯文本。在 LangGraph 中通过定义 Pydantic `BaseModel` 并传入 `response_format` 参数实现。Agent 的输出可通过 `response['structured_response']` 获取。

4. **三种流式输出模式：**
   - `stream_mode="updates"`：每个节点完成后推送该节点的完整更新
   - `stream_mode="messages"`：逐 token 推送 LLM 生成的文本内容
   - `stream_mode="values"`：每步推送 State 的完整快照

5. **`ToolRuntime` 的作用**：在工具调用时注入运行时上下文（如用户角色、认证信息等）。工具通过 `runtime.context` 获取上下文数据，自主判断是否具备执行权限。若权限不足可抛出 `PermissionError`。

6. **`bind_tools` vs `tools` 参数**：
   - `bind_tools` 是 LLM 实例的方法，将工具绑定到模型使其知道如何调用
   - `create_agent(tools=...)` 是高层封装，内部自动完成工具绑定和 ToolNode 配置
   - 使用 `StateGraph` 手动构建时需用 `bind_tools`；使用 `create_agent` 快捷方式时用 `tools` 参数

### 进阶题参考答案

1. **ReAct Agent vs 传统 Chain：**
   - **Chain**：线性执行，每个步骤预先定义，无分支和循环
   - **ReAct Agent**：循环执行，根据中间结果自主决策下一步，支持工具调用
   - **选择场景**：当业务流程不确定、需要动态决策时用 ReAct Agent；当流程固定、步骤明确时用 Chain

2. **细粒度权限控制方案：** 可以在 Context 中添加 `user_id`、`permissions`（权限列表）、`resource_limits`（资源限制）等字段。工具内部根据 `runtime.context.permissions` 检查当前操作是否被允许，并可以通过 `runtime.context.resource_limits` 限制调用次数或频率。

3. **updates 与 messages 模式差异：** `updates` 模式在每个节点执行完毕后产生一次更新，粒度较粗但信息完整；`messages` 模式在 LLM 生成每个 token 时推送，粒度细但只包含文本内容。选择原则：如果关注 LLM 生成过程（如打字机效果）用 `messages`；如果关注节点间的状态流转用 `updates`。

### 编程题参考实现

```python
from typing import Literal, Any
from pydantic import BaseModel, Field
from langchain.tools import tool, ToolRuntime
from langchain.agents import create_agent
from langchain_openai import ChatOpenAI
import os
from dotenv import load_dotenv

_ = load_dotenv()

llm = ChatOpenAI(
    model="qwen3-coder-plus",
    api_key=os.getenv("DASHSCOPE_API_KEY"),
    base_url=os.getenv("DASHSCOPE_BASE_URL"),
)

# 1. 定义 Context 和权限模型
class Context(BaseModel):
    authority: Literal["admin", "user"]

# 2. 定义工具函数
@tool
def get_time(runtime: ToolRuntime[Context, Any], city: str) -> str:
    """获取指定城市当前时间"""
    if runtime.context.authority != "admin":
        raise PermissionError("普通用户无法使用此工具")
    return f"{city}的当前时间是 12:00 PM"

@tool
def search_wiki(runtime: ToolRuntime[Context, Any], query: str) -> str:
    """搜索维基百科"""
    return f"关于'{query}'的搜索结果：..."

# 3. 定义输出格式
class AgentResponse(BaseModel):
    answer: str = Field(description="回答内容")
    tools_used: list[str] = Field(description="使用的工具列表")

# 4. 创建 Agent
agent = create_agent(
    model=llm,
    tools=[get_time, search_wiki],
    response_format=AgentResponse,
    system_prompt="You are a helpful assistant.",
)

# 5. 测试 admin 用户
response = agent.invoke(
    {"messages": [{"role": "user", "content": "现在东京几点了？"}]},
    config={"configurable": {"thread_id": "1"}},
    context=Context(authority="admin"),
)
print(response['structured_response'])
```

### 大厂面试题参考解答

1. **【百度】State 共享的重要性：** 生产级 Agent 需要在整个工作流中持续维护状态（State），而不是孤立的 Prompt。因为 Agent 涉及多轮对话、工具调用、分支决策等复杂流程，共享状态可以保持上下文一致性，支持中断恢复（checkpoint），以及实现记忆功能。

2. **【大厂通用】LangChain 的核心定位：** 是一个专门开发大语言模型应用的框架。主要解决三个问题：
   - **标准化封装**：统一不同模型、向量库、工具提供商的接口
   - **组件化编排**：提供 Chain、Agent、Tool 等可组合的抽象
   - **状态管理**：通过 LangGraph 支持有状态、多步骤的工作流

3. **【阿里】ReAct 循环机制：** LLM 接收系统提示词和对话历史作为输入，输出包含推理过程和行动指令。如果输出包含工具调用，Agent 执行工具并将结果作为观察加入上下文；LLM 再基于更新后的上下文继续推理。关键技术点是将工具的 Schema 作为函数描述注入 LLM 的上下文，使 LLM 理解何时调用、如何调用工具。

4. **【字节跳动】`create_agent` vs `StateGraph`：**
   - `create_agent`：高层封装，自动处理 ReAct 循环、工具节点、条件边等，适合快速构建标准 Agent
   - `StateGraph`：底层手动控制，需要手动定义节点、边、条件路由，适合需要精细控制流程的业务
   - 选择：标准问答/工具调用场景用 `create_agent`；需要复杂分支、人机交互、多步骤流水线时用 `StateGraph`
